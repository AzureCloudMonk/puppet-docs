Puppet Enterprise (PE) instala muchos componentes de software, ficheros de configuración, bases de datos, servicios, usuarios y ficheros de log. Es útil conocer la localización de los mismos, en caso de solucionar problemas o gestionar tu infraestructura PE.

PE instala bastante componentes de software y dependencias.

Los componentes funcionales de PE estas están separado entre los empaquetados con puppet-agent y los empaquetados en el lado del server (el cual también incluye puppet-agent).

Binarios, modules y plugins instalados

PE instala muchos binarios, módulos y plugins para operaciones normales y para interactivos con sus herramientas y servicios.

Binarios ejecutables y enlaces simbólicos

PE instala binarios ejecutables para interactuar con herramientas y servicios.

En nodos *nix, todo el software PE está instalado bajo /opt/puppetlabs.

En nodos Windows, todo el software PE está instalado en el subdirectorio "Puppet Enterprise" del directorio de aplicaciones 32-bit estandar

Binarios ejecutables en *unix esta en /opt/puppetlabs/bin y /opt/puppetlabs/sbin.

Para hacer disponibles herramientas esenciales de Puppet para todos los usuarios, el instalador crea automáticamente enlaces simbólicos en /usr/local/bin para los binarios facter, puppet, pe-man, r10k, hiera y mco. Nota que los enlaces simbólicos solo serán creados si /usr/local/bin es grabable.

AIX y Solaris necesita agregar 10/11 usuarios /usr/local/bin a su ruta predeterminada.

Si estás ejecutando agents Mac OS X, nota que enlaces simbólicos no están creados hasta la primera ejecución completa de Puppet que aplica los catálogos de agent.

Los binarios provistos por otros componentes PE, aquellos que interactúan con el servidor PostgreSQL instalado de PE, PuppetDB o los paquetes de Ruby no tienen enlaces simbólicos creados.

Para obtener instrucciones sobre cómo habilitar binarios o deshabilitar enlaces simbólicos, consulte lo siguiente:

  • Agrege los bianrios de Puppet Enterprise al PATH
  • Deshabilitar enlaces simbólicos Puppet Enterprise

Módulos y plugins

PE instala algunos módulos y plugins para operaciones normales.

  • Los modules de Puppet incluidos en PE están instalados en el servidor Puppet master en /opt/puppetlabs/puppet/modules. No realizes ningún cambio o agregues modules por tu cuenta. En su lugar instalelos en /etc/puppetlabs/code/environments/<environment>/modules.
  • Los plugins de MCollective están instalados en /opt/puppetlabs/mcollective/plugins/ *nix y en <COMMON_APPDATA>\PuppetLabs\mcollective\etc\plugins\mcollective sobre Windows. Si tu estás agregando nuevos plugins en los nodos PE agent , tu deberías distribuirlos vía Puppet como está descrito en la página del manual “Adding actions”.

FIcheros de configuración instalados

PE instala configuración de ficheros, los cuales, de vez en cuando, puedes necesitar para interactuar con ellos.

Sobre nodos *nix, todos los ficheros de configuración de Puppet Enterprise se encuentran en /etc/puppetlabs/puppet.

Sobre nodos Windows, todos los fichero de configuración se encuentran bajo <COMMON_APPDATA>\PuppetLabs. La localización de la carpeta varía según la versión de Windows; en las versiones 2008 y 2012, la localización por defecto es C:\ProgramData\PuppetLabs\puppet\etc.

El confdir de Puppet se encuentra en el subdirectorio puppet. El directorio contiene el fichero puppet.conf, auth.conf y el directorio SSL.

Utilidades instaladas

PE instala múltiples suites de herramientas para ayudar en el trabajo con los componentes principales del software.

Esas herramientas incluyen:

  • Puppet Tools — Herramientas para controlar las funciones básicas de Puppet como puppet master y puppet cert. Consulta la Sección Herramientas del Munual Puppet para más información.
  • PE client tools — El paquete pe-client-tools es un conjunto de utilidades CLI que amplían las capacidades para acceder a los servicios de Puppet Enterprise desde el Puppet master o una workstation. Este paquete incluye:
    • Puppet orchestrator — El Puppet orchestrator es un conjunto de utilidades de línea de comando que proveen de una interfaz al servicio Puppet Application, y también provee de la capacidad de aplicar cambios a nivel de entorno. Estas herramientas incluye puppet job y puppet app. Consulta la documentación de Puppet Orchestrator para más información. Consulta la documentación de Code Manager para más información.
    • Puppet Access — Los escenarios de Puppet Enterprise pueden generar tokens para autenticar su acceso a ciertos comandos de PE y en las distintas APIs . Consulte la documentación en autenticación basada en token para más información.
    • Code Manager CLI — El comando puppet code Comando le permite activar el Code Manager desde la línea de comandos para implementar su entorno.
    • PuppetDB CLI — Esta es una utilidad para trabajar con PuppetDB, como realizar consultar y exportaciones de datos. Consulta PuppetDB CLI para más información.
  • Utilidades MCollective — Herramientas para invocar acciones simultáneas entre un número de nodos. Estas utilidades están construidas en el MCollective framework y son accesibles desde el comando mco. Consulta la Documentación PE MCollective para más información.
  • Utilidades Module — La utilidad Modules es utilizada para acceder y crear Puppet Modules, is used to access and create Puppet Modules, que son trozos reutilizables de código escrito por los usuarios de Puppet para automatizar tareas de configuración y despliegue. Para más información y consultar los modules, visite Puppet Forge.
  • Console — La consola web UI de Puppet Enterprise. La consola provee de herramientas para consultar y editar los recursos en sus nodos, ver reportes, mostrar la actividad mediante gráficas, etc. Consulte la sección de consola del Manual PE para más información.

BBDD instaladas

PE instala múltiples BBDD predeterminadas, todas las cuales utilizan PostgreSQL como un backend de base de datos.

La BBDD PostgreSQL de PE incluye las siguientes bases de datos:

Base de Datos Descripción
pe-activity Datos de actividad del Classifier, incluyendo quien, que y cuando
pe-classifier Datos de clasificación, información de todos los grupos de nodos
pe-puppetdb Datos PuppetDB, incluyendo recursos exportados, catálogos, facts y informes.
pe-rbac Datos RBAC, incluyendo usuarios, permisos y información AD/LDAP
pe-orchestrator Datos del orchestrator, incluyendo detalles sobre trabajos en ejecución (usuarios, nodos y salida de la ejecución)

Utilice herramientas nativas PostgreSQL para exportar e importar BBDD. Como mínimo, debes realizarla todas las noches el backup a un sistema remoto o como dicta la política de su empresa.

Ficheros log instalados

El software distribuido por Puppet Enterprise genera ficheros de log que usted puede consultar o utilizar para solucionar problemas.

Puppet master logs

El Puppet master tiene los siguientes logs.

  • /var/log/puppetlabs/puppetserver/puppetserver.log: La aplicación de logs de Puppet esta activa. Aquí es donde se pueden encontrar cosas como errores de compilación y advertencias de obsolescencia.
  • /var/log/puppetlabs/puppetserver/puppetserver-daemon.log:es donde puede encontrar fatal errors o crash reports.
  • /var/log/puppetlabs/puppetserver/pcp-broker.log: El archivo de registro para los intermediarios de PCP en los masters de compilación.
  • /var/log/puppetlabs/puppetserver/code-manager-access.log
  • /var/log/puppetlabs/puppetserver/file-sync-access.log
  • /var/log/puppetlabs/puppetserver/masterhttp.log
  • /var/log/puppetlabs/puppetserver/puppetserver-access.log
  • /var/log/puppetlabs/puppetserver/puppetserver.log
  • /var/log/puppetlabs/puppetserver/puppetserver-status.log

Puppet agent logs

La localización de los logs de Puppet agent depende del sistema operativo del agent.

En nodos *nix, la actividad del servicio de logs de Puppet agent va al servicio syslog. La configuración de syslog determina dónde se guardarán estos mensajes, pero la ubicación predeterminada es /var/log/messages en Linux, /var/log/system.log en Mac OS X y /var/adm/messages en Solaris.

En nodos Windows, la actividad del servicio de logs de Puppet agent va al Windows Event Log. Usted puede consultar estos logs mediante el navegador del Visor de Eventos. (Panel de Control > Sistema y Seguridad > Herramientas Administrativas > Visor de Eventos)

ActiveMQ logs

ActiveMQ tiene los siguientes logs.

  • /var/log/puppetlabs/activemq/wrapper.log
  • /var/log/puppetlabs/activemq/activemq.log
  • /var/log/puppetlabs/activemq/data/audit.log

MCollective logs

MCollective tiene los siguientes logs.

  • /var/log/puppetlabs/mcollective.log: mantenido por el servicio MCollective, el cual está instalado en todos los nodos.
  • /var/log/puppetlabs/mcollective-audit.log: existe en todos los nodos que tiene instalado MCollective; realiza un log sobre las acciones ejecutadas en el nodo por MCollective, incluyendo información sobre el cliente que llamó al nodo

Logs de Consola y pe-console-services

La consola y pe-console-services tiene los siguientes logs.

  • /var/log/puppetlabs/nginx/error.log: contiene errores relacionados con nginx. Los errores de consola que no se registran en ningún otro lugar se pueden encontrar en este registro. Si tiene problemas con la consola o Puppet, este registro puede ser útil.
  • /var/log/puppetlabs/nginx/access.log
  • /var/log/puppetlabs/console-services.log
  • /var/log/puppetlabs/console-services-access.log
  • /var/log/puppetlabs/console-services/console-services-api-access.log
  • /var/log/puppetlabs/console-services-daemon.log: tSu es donde se pueden encontrar errores fatales o informes de fallos.

Installer logs

El instalador de PE tiene los siguientes logs.

  • /var/log/puppetlabs/installer/http.log: Contiene las solicitudes web enviadas al instalador; Presente únicamente en la máquina desde la que se realizó la instalación basada en web
  • /var/log/puppetlabs/installer/installer-<timestamp>.log: Contiene las operaciones realizadas y los errores que ocurrieron durante la instalación
  • /var/log/puppetlabs/installer/install_log.lastrun.<hostname>.log: Contiene el contenido de la última ejecución del instalador

Certificados instalados

Durante la instalación, PEe genera e instala un número de certificados SSL para que agents y servicios puedan autenticarse.

Estos certificados puedes ser encontrados en /etc/puppetlabs/puppet/ssl/certs.

Certificados Definición
<PUPPET MASTER CERTNAME> Generados durante la instalación. En una instalación monolítica, este certificado es utilizado por PuppetDB y la consola PE. Este es el mismo valor para el certname de los agents que se ejecuta en Puppet master. En instalación monolítica, el agent en la consola PE y PuppetDB comparten el certname. En un instalación dividida o monolítica por defecto, esto también es el Puppet CA cert.
<PE CONSOLE CERTNAME> El certificado para la consola PE, el cual es únicamente generado, si usted tiene una instalación dividida. Este es el mismo valor del certname de los agents que ejecuta en una consola PE.
<PUPPETDB CERTNAME> El certificado para PuppetDB, el cual es generado únicamente si tiene una instalación dividida. Este es el mismo valor para el certname de los agents que ejecutan en PuppetDB.
pe-internal-mcollective-servers Un certificado generado en un Puppet master y ser comparte con todos los nodos agentes.
pe-internal-peadmin-mcollective-client El certificado para la cuenta peadmin en el Puppet master.
pe-internal-puppet-console-mcollective-client El certificado MCollective para la consola PE.

Servicios que se ejecutan en el Puppet master o en la consola (por ejemplo, pe-orchestration-services y pe-console-services, utiliza el certificado de Puppet agent para autenticar.
